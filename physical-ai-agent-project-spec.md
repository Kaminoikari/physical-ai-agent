# Physical AI Side Project Spec
## 用 LLM Agent 調度有限技能，完成沒見過的任務（純模擬版）

> 版本：v1.0 ｜ 定位：Side Project 執行藍圖 ｜ 目標讀者：你自己（執行）＋ 面試官（敘事）
> 核心定位：**底層動作交給現成模型，價值放在 agent 的任務拆解與閉環回饋（PM 與系統設計的主場）**

---

## 0. 一句話定義

打造一個純模擬的具身 agent 系統：使用者用自然語言下一個**沒有預先寫死流程**的任務，agent 將其拆解成一連串**已訓練好的底層技能**（抓、放），自主執行，並透過視覺回饋判斷成敗、必要時重試。

**使用者體感到的**：「我講一句話，虛擬手臂就完成了一件我沒明確教過的任務。」
**底層實際在做的**：LLM agent 把模糊人話翻譯成明確的技能呼叫序列，調度有限的預訓練 policy。

---

## 1. 為什麼是這個目標（先對齊心態）

- **不是退而求其次**：底層動作訓練是工程師的活；「如何用 agent 把有限能力組合成完成任意任務」是產品與系統設計的活，正是 PM 的差異化主場。
- **與頂級玩家同源**：NVIDIA 的架構（LLM/VLM 調度預訓練的底層運動模型）、MetAI 的工業場景（有限標準動作組合成多變任務），本質都是這套。你做的是正確架構的縮小版，不是玩具。
- **務實邊界（牢記）**：agent 能組合的，永遠只限於底層有的技能。只訓練「抓/放」，就只能完成「抓放的排列組合」任務（分類、堆疊、搬運、排序）。這個範圍比想像廣，但不是無限。

---

## 2. 三層架構

```
┌─────────────────────────────────────────────┐
│  L3 — Agent 編排層（你的主場，最高價值）          │
│  - 自然語言理解 + 任務拆解                        │
│  - 技能調度決策                                  │
│  - 閉環回饋：看畫面判斷成敗 → 重試/調整            │
│  用現成大模型（Claude / GPT），靠 prompt，不訓練     │
├─────────────────────────────────────────────┤
│  L2 — 技能介面層（黏合膠）                        │
│  - 把每個 policy 包成 agent 可呼叫的 function     │
│  - pick(obj) / place(target) / 視覺查詢 query()  │
├─────────────────────────────────────────────┤
│  L1 — 技能執行層（用現成的，不戀戰）               │
│  - SmolVLA base 在模擬環境執行「抓/放」            │
│  - 跑在 LeRobot 整合的模擬環境裡                  │
└─────────────────────────────────────────────┘
```

**設計鐵則：讓 L3 與 L2 完全不知道 L1 是模擬還是真實。** 這樣未來要接真實硬體，只換 L1，上層不重寫。這是你低成本先做完最難 80% 的關鍵。

---

## 3. 技術選型（與理由）

| 層 | 選型 | 理由 |
|----|------|------|
| 模擬環境 | LeRobot 整合的模擬（LIBERO / Meta-World 類） | 與 SmolVLA 同生態、官方支援、可直接載入評估 |
| 底層 policy | `lerobot/smolvla_base`（450M VLA） | 官方、輕量、可在單張消費級 GPU 甚至 CPU 跑；官方範例就是 SO-101 揀放 |
| 配套資料 | `lerobot/svla_so101_pickplace` 或模擬內建資料 | 官方欽定、格式保證相容 |
| Agent 框架 | LLM agent（Claude 為大腦）+ 工具呼叫 | 你天天用 Claude Code 的主場；可參考 RoboCrew 的「VLA 當工具」設計哲學 |
| 視覺回饋 | VLM（看模擬畫面判斷成敗） | 閉環的命脈；模擬畫面乾淨，判斷比真實世界容易 |
| 算力 | 雲端 GPU（L4 / A100 等級） | 訓練/微調約數小時；無需自購卡 |

---

## 4. 範圍界定（務必先讀）

### ✅ In Scope（要做）
- 2 個底層技能：`pick`、`place`
- Agent 把自然語言任務拆解成技能序列
- 視覺閉環：每步執行後確認成敗，失敗可重試
- Demo「臨場、未寫死」的指令
- 一篇拆解文 + 一段 demo 影片

### ❌ Out of Scope（明確不做，避免時間黑洞）
- 不做「沒訓練過的新動作」（這是全領域聖杯，無人能解）
- 不追求高成功率完美主義（六成上下即可 demo）
- 不碰真實硬體（本版純模擬）
- 不自己從零訓練 agent（用現成大模型）
- 不做多於 2 個的技能（先把抓放組合做透）

## 5. 逐週里程碑（每週投入 10–15 小時，總 5–6 週）

### Week 1 — 環境跑通 + 看懂現成資產
**交付物**：能在本機/雲端載入 LeRobot 模擬環境，成功載入 `smolvla_base`，跑通官方範例（虛擬手臂在模擬裡動起來）。
**高風險卡點**：環境依賴（ffmpeg、PyTorch/TorchCodec 版本相容）。**對策**：用官方提供的 GPU Docker image 或 Colab Notebook，不要自己從零裝環境。
**自我檢核**：能不能在模擬裡看到手臂執行一次「抓」？不行就先別往下。

### Week 2 — 底層技能就緒
**交付物**：在模擬裡有可靠的 `pick` 與 `place` 兩個技能（用 smolvla_base，必要時用官方資料微調）。每個技能單獨呼叫時成功率堪用（≥60%）。
**心法**：**先用官方資料把整條「載入→執行→評估」管線打通，再考慮換自己的資料。** 不要太早改架構，先讓預設設定正常運作。
**高風險卡點**：技能成功率太低。**對策**：先確認是資料問題還是設定問題（這就是先用官方資料的價值——排除變因）。

### Week 3 — 技能介面層 + Agent 骨架
**交付物**：把 `pick`/`place` 包成 agent 可呼叫的 function；agent 能接收一句指令、輸出一個技能呼叫序列（先不接視覺，先讓「拆解」對）。
**範例**：使用者說「把紅色方塊放到 A 區」→ agent 輸出 `[pick(red_cube), place(zone_A)]`。
**這是你的主場**：prompt 設計、工具描述、拆解邏輯，做得漂亮。

### Week 4 — 閉環回饋（成敗關鍵）
**交付物**：接上 VLM 視覺回饋。每執行一步，agent 看模擬畫面確認「成功了沒」；失敗則重試或調整。
**範例**：抓取失敗 → VLM 回報「方塊還在原位」→ agent 重試 `pick`。
**高風險卡點**：這層最難，是整個系統「看起來聰明」的命脈。**對策**：先做最簡單的二元判斷（成功/失敗），不要一開始就追複雜的場景理解。

### Week 5 — Demo 設計 + 亮點橋段
**交付物**：完整 demo 腳本與錄影。必含兩個質感跳級橋段：
1. **臨場未寫死指令**：demo 時下一個你沒寫過流程的任務（如「紅色放左、藍色放右」），agent 當場拆解執行 → 證明是真理解，不是腳本。
2. **失敗自我修正**：某次抓取失敗，agent 透過視覺察覺並重試 → 展示 embodied agent 最核心能力。

### Week 6 — 拆解文 + 收尾
**交付物**：一篇繁中深度拆解文（大綱見 §7）+ 整理 GitHub repo + demo 影片。
**Buffer**：這週也當作前面任一週 overrun 的緩衝。

## 6. 可量測成功指標

| 指標 | 最低標 | 理想 |
|------|--------|------|
| 底層技能單獨成功率 | ≥60% | ≥80% |
| Agent 任務拆解正確率（拆對序列） | ≥80% | ≥95% |
| 端到端任務完成率（含重試） | ≥50% | ≥70% |
| 能處理的「未寫死」任務種類 | ≥3 種 | ≥6 種 |
| 失敗重試機制是否運作 | 能 demo 一次 | 穩定觸發 |

> 注意：六成上下的完成率就足夠 demo。失敗片段是拆解文的好素材，不是恥辱。

## 7. 拆解文大綱（你接觸 MetAI 最有效的敲門磚）

**標題方向**：〈當機器人只會「抓」和「放」：我用 LLM Agent 讓有限技能完成無限任務的實驗〉

1. **開場——產業問題**：Physical AI 為什麼難規模化？因為不可能為每個新任務都重訓一個模型。引出「上層智慧調度」的必要性。
2. **架構——三層拆解**：用 NVIDIA「LLM 調度預訓練運動模型」的公開架構當參照，說明你的縮小版。展示你站在正確的學術/工程脈絡。
3. **實作——我做了什麼**：誠實呈現 agent 拆解 + 視覺閉環。放成功與失敗的對比。
4. **為什麼先用模擬**：點明這是 NVIDIA、MetAI 的標準工作流（先在模擬大量試錯）。把「模擬」講成「正確的工程順序」而非「廉價版」。
5. **下一步——sim-to-real 與 world model**：明確指出 sim-to-real 鴻溝是下一關，正是 MetAI 在解的問題；並提及 world model 是更前面的方向。**展示你看得比多數人遠。**

**對 MetAI 的隱藏作用**：MetAI 想在台灣建立 Physical AI 話語權，一篇高品質繁中拆解剛好餵養它要的市場教育。發出去 tag 對的人，比投履歷有效。

## 8. 面試論述彈藥（基於這個 project）

- **「為什麼工業 Physical AI 一定要上層調度？」**：因為客戶不可能為每個新任務重訓模型，必須靠有限標準動作的智慧組合——你親手做過這個調度層。
- **「sim-to-real 你怎麼看？」**：你刻意先用模擬驗證 agent 架構（正確工程順序），並清楚 sim-to-real 是下一關、為何關鍵——展示判斷力 ≈ 親手跨過它。
- **「你跟工程師的差異？」**：工程師做底層 policy；你做「把有限能力翻譯成完成任意任務」的系統與產品設計，並用你的 B2B SaaS / 企業客戶 domain 連接到真實工業需求。

---

## 9. 未來升級路徑（可選，非必做）

因為 L3/L2 與 L1 解耦，未來若要接真實硬體：
1. 買組裝好的 SO-101（台灣現貨：機器人王國「組裝完成＋雙相機」組合）
2. 只替換 L1（模擬 → 真實 policy），上層幾乎不動
3. 新增的唯一大工作：面對 sim-to-real gap（而這正好變成拆解文 Part 2 的真實素材，且直擊 MetAI 核心）

> 你等於用低成本先做完最難的 80%（agent 架構），硬體是可選的「最後升級」。

## 10. 立即可開始的第一步

1. 申請一個雲端 GPU 環境（或先用官方 Colab Notebook 零成本起步）
2. `git clone` LeRobot，安裝 smolvla 相依
3. 載入 `lerobot/smolvla_base`，跑通官方模擬範例
4. 確認「虛擬手臂在模擬裡動起來」——這是 Week 1 的成功訊號

---

### 誠實提醒（這份 spec 最弱的一環）

**Week 4 的視覺閉環是整個 project 風險最高、也最可能拖時間的地方。** 它是「看起來聰明」的命脈，但 VLM 判斷模擬畫面成敗的可靠度，可能需要反覆調 prompt 與相機視角。

**強化建議**：把 Week 4 的成功標準先壓到最低（只要能做出「一次」成功的失敗偵測 + 重試即可 demo），不要追求穩定。穩定性留到有餘力再補。若 Week 4 嚴重卡關，退路是：拆解任務時讓 agent「假設每步成功」先跑通端到端，把視覺閉環降級為「加分項」而非「必要項」——這樣至少保住端到端 demo 不開天窗。
