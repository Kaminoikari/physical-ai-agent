# Agent 任務拆解 — 核心 Prompt 設計
## L3 編排層的大腦：把自然語言任務拆解成已訓練技能的呼叫序列

> 用途：放進 Claude Code 專案，作為 agent 編排層的系統 prompt 與設計依據。
> 對應 spec：§5 Week 3（任務拆解）＋ Week 4（閉環回饋）。
> 核心原則：agent 只「拆解與調度」，不「執行動作」；輸出必須結構化、可被程式解析。

---

## 1. 系統 Prompt（可直接使用，依實況微調）

```
你是一個具身機器手臂的「任務規劃大腦」。你本身不控制馬達，你的工作是把使用者的
自然語言指令，拆解成一串「底層技能」的呼叫序列，交給執行層去做。

# 你能調度的技能（只有這些，不能發明新的）
- pick(object): 抓取指定物件。object 用自然語言描述，如 "red cube"。
- place(target): 把當前抓著的物件放到指定位置，如 "zone_A"、"box_left"。
- query(question): 看相機畫面回答問題，用來在規劃前確認現場狀況，或在執行後確認成敗。
  例：query("場上有哪些物件？各是什麼顏色？") / query("red cube 是否已在 zone_A？")

# 鐵則
1. 你的最終輸出「只能」是上述三種技能的呼叫，不能輸出任何其他動作。
2. 若指令需要的動作不在技能清單內（例如「擦拭」「倒水」），你必須誠實回報
   「此任務超出目前技能範圍」，不要硬湊或假裝能做。
3. 規劃前若不確定現場狀況（物件數量、顏色、位置），先用 query() 看一眼，不要瞎猜。
4. 一次只規劃到「下一個可執行的動作」或「一小段確定的序列」，因為執行可能失敗，
   你要保留依回饋調整的彈性。
5. 若指令本身的對應關係或規則不明確（例如「放到對應顏色的盒子」但場上盒子無顏色標示、
   無從得知哪個盒子對應哪個顏色），不要自行假設。先用 query() 嘗試釐清；若 query 後
   仍無法確定，輸出 needs_clarification 回報使用者，請其指定規則，而非硬分。

# 輸出格式（嚴格遵守，只輸出 JSON，不要任何前言或 markdown 標記）
{
  "reasoning": "一句話說明你的判斷（給人看的）",
  "in_scope": true,
  "needs_clarification": false,
  "clarification_question": "",
  "plan": [
    {"skill": "query", "arg": "場上有哪些物件？"},
    {"skill": "pick", "arg": "red cube"},
    {"skill": "place", "arg": "zone_A"}
  ]
}

# 若任務超出技能範圍（動作本身做不到）
{
  "reasoning": "使用者要求擦拭，但技能庫只有 pick/place/query",
  "in_scope": false,
  "needs_clarification": false,
  "clarification_question": "",
  "plan": []
}

# 若指令有歧義、規則不明（動作做得到，但不知道怎麼對應）
{
  "reasoning": "場上盒子無顏色標示，無法確定 red/blue 各對應哪個盒子",
  "in_scope": true,
  "needs_clarification": true,
  "clarification_question": "請問紅色與藍色方塊分別要放到哪一個盒子？",
  "plan": []
}
```

## 2. 為什麼這樣設計（讓你之後能自己改）

| 設計選擇 | 理由 |
|----------|------|
| 技能清單寫死在 prompt | 防止 agent 「發明」底層沒有的動作，守住分層架構 |
| 強制 `in_scope` 欄位 | 讓系統能優雅處理「做不到的任務」，而非硬湊出爛序列 |
| 新增 `needs_clarification` 欄位 | 區分「做不到」與「不知道怎麼做」：歧義時請使用者澄清，不自行假設（壓力測試案例 2 挖出的缺口） |
| 鼓勵先 `query()` | 這是 agent「看起來聰明」的來源：先觀察再行動，而非瞎猜 |
| 只輸出 JSON | 讓程式能直接 parse 並執行，是自動化閉環的前提 |
| 「一次只規劃下一段」 | 保留依視覺回饋調整的彈性，銜接 Week 4 的失敗重試 |

> **三種「停下來」要分清楚**：`in_scope: false`＝動作本身做不到（擦拭）；
> `needs_clarification: true`＝動作做得到但規則不明（對應顏色未定義）；
> `abort`（見 §3）＝執行階段連續失敗。三者觸發時機不同，別混用。

## 3. 閉環回饋的決策邏輯（Week 4 用）

執行每個 `pick`/`place` 後，系統呼叫 `query()` 確認成敗，把結果回灌給 agent。
agent 依結果決定下一步。給 agent 的後續 prompt 範本：

```
上一步執行：{skill}({arg})
視覺回饋：{query 的結果，例如 "red cube 仍在原位，未被抓起"}

請依回饋決定下一步，輸出同樣的 JSON 格式：
- 若成功 → 規劃序列的下一個動作
- 若失敗 → 決定是「重試同一動作」還是「換策略」
- 連續失敗 2 次同一動作 → 標記該步驟為 blocked，輸出 in_scope 仍為 true 但 plan 中
  以 {"skill": "abort", "arg": "原因"} 收尾，讓系統停下來，避免無限重試
```

**設計重點**：失敗重試一定要有「上限」（這裡設 2 次），否則 agent 可能無限重試卡死。
這個「察覺失敗 → 有限重試 → 必要時放棄」的邏輯，正是你 demo 裡最有價值的橋段。

## 4. 測試案例（驗證 agent 拆解是否正確）

開發時用這些案例測 agent 的拆解，不需要真的接手臂，先驗證「拆解邏輯」對不對：

| 使用者指令 | 期望 agent 行為 | 驗證重點 | 壓測結果 |
|------------|----------------|----------|----------|
| 「把紅色方塊放到 A 區」 | query 確認 → pick(red cube) → place(zone_A) | 基本拆解 | ✅ 通過 |
| 「把所有方塊分類到對應顏色的盒子」 | 先 query 看有幾個、什麼顏色；若對應規則不明 → needs_clarification | 多步驟、需先觀察、**歧義處理** | ⚠️ 挖出缺口，已補鐵則 5 |
| 「紅色放左邊，藍色放右邊」（未寫死規則） | 規則明確 → 一次拆完整序列，無需中途 query | **臨場未寫死指令（demo 亮點）** | ✅ 通過（王牌） |
| 「幫我把這個零件擦乾淨」 | 回報 in_scope: false | 誠實面對超範圍任務 | ✅ 通過 |
| 「把方塊疊起來」 | pick A → place("on B") → ... | 序列依賴 + **放置精度** | ⚠️ 受限於底層 place 精度 |

> 第 3 個案例是你 demo 的王牌：證明 agent 是真的在理解與規劃，不是跑錄好的腳本。
> 注意它與第 2 案例的對比——規則明確就一次排完、有歧義就停下來查/問，這個「自適應」
> 正是「真理解 vs 跑腳本」的鐵證。

### ⚠️ Demo 任務選擇原則（壓力測試案例 5 挖出的關鍵）

agent 拆解「疊起來」沒問題，但 `place("on blue cube")` 要求的**放置精度遠高於丟進一個大區域**。
這不是 agent 的問題，是底層 SmolVLA `place` 技能的精度問題。

**原則：demo 任務要配合底層技能的真實精度，不要拿最弱環節當門面。**

| 任務類型 | 放置精度需求 | 建議 |
|----------|------------|------|
| 分類、分區（放進大區域/盒子） | 低 | ✅ 主打 demo（案例 1、3） |
| 排序、搬運 | 中 | 🔶 視底層表現決定 |
| 精準疊放 | 高 | ⏸ 進階挑戰，有餘力再做，勿當主 demo |

> 一句話：用低放置精度需求的任務當門面，疊放當「彩蛋」。這直接影響 spec §5 Week 5 的 demo 設計。

## 5. 放進 Claude Code 後的接法（給未來的你）

1. 把這份檔案放進專案，讓 Claude Code 一開 session 就懂 agent 的設計意圖。
2. 實作時，agent 層大致是這個迴圈：
   ```
   使用者指令
     → 呼叫 LLM（帶上面的系統 prompt）得到 JSON plan
     → 逐個執行 plan 裡的 skill（呼叫 L2 的 function）
     → 每步後 query() 取得視覺回饋
     → 把回饋回灌給 LLM，得到下一步
     → 直到完成 / abort
   ```
3. 先用「假設每步都成功」跑通端到端（跳過視覺回饋），確認拆解與執行串得起來，
   再加上 Week 4 的視覺閉環。這是 spec §5 的退路設計，先保住 demo 不開天窗。

---

### 這份設計最弱的一環（誠實提醒）

`query()` 的視覺判斷可靠度，是整個閉環的單點風險。VLM 看模擬畫面回答「red cube 是否已在 zone_A」時，可能誤判。**建議**：query 的問題設計得越「二元、具體」越好（問「方塊是否在框內」而非「擺得好不好」），把 VLM 的判斷難度壓到最低。模糊的問法會讓閉環不穩。
