# query() 視覺判斷優化設計
## 把 VLM 的判斷難度壓到最低，讓閉環回饋可靠

> 用途：放進 Claude Code 專案，作為 L3 閉環回饋層 query() 的設計依據。
> 對應 spec：§5 Week 4（閉環回饋）＋ agent prompt 檔的「最弱一環」。
> 核心問題：query() 的視覺判斷是整個閉環的單點風險。VLM 出錯多半不是「看不到」，
> 而是被要求做「主觀判斷」或「精確空間推理」——這兩件事 VLM 都很弱。

---

## 1. 核心原則

**每一次 query 都要讓 VLM 只需回答「是 / 否」，並消除空間推理與主觀評價成分。**

| ❌ 不要這樣問 | 為什麼爛 | ✅ 改成這樣問 |
|--------------|----------|--------------|
| 「方塊擺得好不好？」 | 主觀判斷，VLM 亂答 | 「red cube 是否完全在紅框內？」 |
| 「red cube 在哪裡？」 | 要空間定位，VLM 不準 | 「red cube 是否在 zone_A？是/否」 |
| 「離框中心幾公分？」 | 要精確量測，VLM 幾乎必錯 | （改用座標判斷，見 §3） |
| 「分類對了嗎？」 | 一次檢查太多東西 | 拆成多個獨立二元問題（見 §2） |

---

## 2. 五個優化技巧

### 技巧 1：一律二元化
永遠問「X 是否成立」，回答限定 yes/no。不問「怎麼樣」「在哪」。

### 技巧 2：拆解，不要複合
與其問一個複雜問題，拆成多個簡單的二元問題，組合判斷整體。
```
❌ query("分類對了嗎？")
✅ query("red cube 是否在 box_left 內？")
   query("blue cube 是否在 box_right 內？")
   → 兩個都 yes 才算成功
```

### 技巧 3：用視覺輔助降低空間推理
在模擬環境給放置區畫明顯的顏色邊框/標籤，問題引用畫面看得見的參照。
```
❌ query("red cube 是否在左邊區域？")   （「左邊」要 VLM 自己推理）
✅ query("red cube 是否在紅色方框內？")  （「紅框」是畫面裡看得見的參照）
```

### 技巧 4：善用模擬特權——讀座標當真值（最重要）
模擬環境知道每個物件的精確座標。**「位置/成敗」這類有客觀標準的判斷，
直接用程式讀座標判斷是否落在目標區域，不靠 VLM。** 這是 100% 準確的 ground truth。
```python
# 偽碼：座標判斷取代視覺判斷
def is_in_zone(obj_pos, zone_bounds):
    return zone_bounds.x_min <= obj_pos.x <= zone_bounds.x_max \
       and zone_bounds.y_min <= obj_pos.y <= zone_bounds.y_max
# 準確、免費、快，因為用的是模擬的真值而非視覺猜測
```

### 技巧 5：VLM 與座標判斷分工
| 判斷類型 | 用什麼 | 理由 |
|----------|--------|------|
| 位置、是否在區域內、成敗 | **座標判斷** | 有客觀標準，真值最可靠 |
| 是否出現新的/未知物件、語意理解 | **VLM** | 真的需要語意，座標給不了 |

---

## 3. 優化後的 query 介面設計

```
query(question, mode):
  - mode="spatial"  → 路由到座標判斷（is_in_zone 等），回傳 true/false
  - mode="semantic" → 路由到 VLM，問二元問題，回傳 yes/no
agent 不需知道底層用哪種，它只管問「X 是否成立」，由 query 內部選最可靠的方式。
```

**好處**：agent 的 prompt 不變（它只管問二元問題），可靠度的工程藏在 query 內部。
未來接真實硬體時，只需把 spatial 模式從「讀座標」換成「VLM 視覺」，上層不動。

---

## 4. ⚠️ Demo 敘事的誠實標示（重要，別跳過）

模擬階段用 ground-truth 座標判斷成敗，是**對的工程選擇**（用最可靠訊號），不是作弊。
但你**必須在拆解文與面試誠實說明**這個設計，否則內行會看穿「閉環其實沒用到視覺」。

**該怎麼說（這反而是加分）**：
> 「在模擬階段，我用 ground-truth 座標做成敗判斷，以確保閉環穩定、隔離變因。
> 但這個訊號在真實世界不可得——真實部署必須靠 VLM 視覺判斷，而視覺判斷的
> 不可靠正是 sim-to-real 的核心難點之一。我的架構刻意把判斷邏輯封裝在 query()
> 介面後，正是為了讓未來從『座標真值』切換到『視覺判斷』時，上層 agent 不需重寫。」

**為什麼這段是子彈**：
1. 展示你誠實、懂自己系統的邊界（PM 的成熟度）
2. 再次精準命中 sim-to-real——在模擬能「作弊」用座標，到真實不能，這落差正是 MetAI 在解的
3. 展示你的封裝設計有前瞻性（為遷移真實硬體預留接口）

## 5. 套用後，閉環回饋長這樣（更新版）

```
執行 pick("red cube")
  → query("red cube 是否已被夾起且離開原位？", mode="spatial")  # 讀座標
  → 若 false：重試（上限 2 次）→ 仍失敗則 abort
執行 place("box_left")
  → query("red cube 是否在 box_left 範圍內？", mode="spatial")   # 讀座標
  → 若 false：重試 → 仍失敗則 abort
（僅在需要語意時才動用 mode="semantic" 的 VLM）
```

---

### 一句話總結
把 query 全部改成二元問題、能用座標判斷就不用 VLM、用視覺輔助消除空間推理；
模擬階段用座標真值確保閉環穩定，並誠實把「真實世界要靠視覺」標示為 sim-to-real 難點。
這樣你的閉環在 demo 時穩，敘事上又精準命中 MetAI 的核心命題。
